## 9.2 Security Control Implementation Details

### Control 1: Mutual TLS Authentication
- **Purpose:** Verify both agent and server legitimacy
- **Implementation:** 
  - Self-signed CA certificate stored in HSM
  - Server certificate issued with 3-year validity
  - Agent certificates issued per-device, unique identifiers
  - Certificate pinning in agent code (prevents MITM via rogue CAs)
  - Certificate rotation every 2 years (automated)
- **Verification:** Agent validates server certificate before connecting

### Control 2: AES-256 Encryption (Transit & At-Rest)
- **Transit Encryption:**
  - TLS 1.3 mandatory (TLS 1.2 disabled)
  - Forward secrecy via ECDHE key exchange
  - 256-bit symmetric cipher suites only
  - HKDF key derivation function
- **At-Rest Encryption:**
  - Database encryption via AES-256-GCM (Galois/Counter Mode)
  - Disk-level encryption via Linux LUKS (Linux Unified Key Setup)
  - Backup encryption via OpenSSL AES-256-CBC
  - Encryption key protection in HSM (no plaintext keys on disk)

### Control 3: Immutable Audit Logs
- **Purpose:** Prevent log tampering; support forensics and compliance audits
- **Implementation:**
  - WORM storage (media prevents overwrite/deletion)
  - Independent from production storage (air-gapped)
  - SHA-256 hash of each log entry
  - Hash chain verification (each entry references previous)
  - Off-site backup of audit logs (geographic diversity)
- **Retention:** 7 years (exceeds GDPR, HIPAA, SOX requirements)

### Control 4: Network Segmentation (VLAN)
- **Purpose:** Isolate RMM from other infrastructure; limit lateral movement
- **Implementation:**
  - RMM on dedicated VLAN (10.50.0.0/24)
  - Firewall rules block RMM-to-other VLAN communication
  - Management access only via VPN from designated admin workstations
  - DMZ for web UI (internet-facing); database on internal VLAN (restricted access)
- **Access Control:** Jump hosts required for database access (no direct connection)

### Control 5: MFA Mandatory (Hardware FIDO2)
- **Purpose:** Prevent credential-based compromise (phishing, password theft)
- **Implementation:**
  - All admin access requires FIDO2 hardware token
  - No software MFA (vulnerability to malware/social engineering)
  - Tokens issued per administrator (unique identifiers)
  - Token list maintained in HSM for offline verification
  - Lost token revocation (automatic in 24 hours)
- **Enrollment:** 50+ authorized administrators issued tokens

### Control 6: EDR Integration
- **Purpose:** Detect behavioral indicators of compromise
- **Implementation:**
  - CrowdStrike Falcon (or Defender for Endpoint) agent on RMM servers
  - Real-time behavioral monitoring (process execution, file access, network connections)
  - Custom rules for RMM-specific suspicious activities
  - Automated response: isolate server on critical detection
  - Forensic data retention: 2 years
- **Alert Escalation:** Critical alerts ‚Üí Page on-call security engineer (5-min response SLA)

### Control 7: Offline Backups (Air-Gapped)
- **Purpose:** Recover from ransomware/sabotage
- **Implementation:**
  - Nightly incremental backups (scheduled 2-4 AM, off-peak)
  - Weekly full backups (Sunday 3 AM)
  - Backups written to offline storage (not connected to network during week)
  - Network connection only for backup write (5-minute window, then disconnect)
  - Testing: Monthly restoration drill (random date, different environment)
- **RTO Target:** <4 hours (restoration from backup)
- **RPO Target:** <1 hour (maximum data loss)

### Control 8: Agent Tamper Detection
- **Purpose:** Verify agent integrity; detect rootkits/malware modifying agent
- **Implementation:**
  - MD5 hash of legitimate agent binary
  - Monthly integrity check (agent compares own binary hash)
  - Hash mismatch triggers alert (SIEM notification, possible quarantine)
  - Code signing: Agent binary signed by Microsoft Authenticode (prevents tampering)
- **Scope:** Windows, macOS, Linux agents all monitored

### Control 9: Certificate Pinning
- **Purpose:** Prevent man-in-the-middle (MITM) attacks via rogue CAs
- **Implementation:**
  - Server public key embedded in agent code (compile-time constant)
  - TLS handshake: Agent verifies server certificate matches pinned public key
  - Mismatch ‚Üí Connection refused, alert generated
  - Update mechanism: New agent version with updated pinned key (gradual rollout)
- **Protection:** Compromised CA cannot issue rogue server certificates

### Control 10: Least Privilege Access (5-Tier Admin Model)
- **Purpose:** Minimize blast radius if admin account compromised
- **Tier Definitions:**
  - **Tier 1 (Read-Only):** View agents, reports (no changes)
  - **Tier 2 (Operator):** Execute tasks (software install, reboot, run scripts) on assigned devices
  - **Tier 3 (Administrator):** Manage admin accounts, policies, custom groups
  - **Tier 4 (Security Administrator):** Access logs, audit trails, compliance reports
  - **Tier 5 (Full Administrator):** All system changes; unrestricted access
- **Assignment:** Principle of least privilege; Tier 5 requires dual-approval from CISO + CTO

# SECTION 10: RISK QUANTIFICATION METHODOLOGY (DETAILED FORMULAS)

## 10.1 FAIR Model Deep-Dive

**FAIR (Factor Analysis of Information Risk) is the quantitative framework underlying the financial analysis:**

### Single Loss Expectancy (SLE)

```
SLE = Asset Value √ó Exposure Factor

Where:
  Asset Value = Replacement/recovery cost of RMM component or data
  Exposure Factor = Percentage of asset value lost in compromise event (0-1.0)

Example 1: RMM Control Server Compromise
  Asset Value = $500K (hardware replacement + staff time to rebuild)
  Exposure Factor = 1.0 (100% compromised, full recovery needed)
  SLE = $500K √ó 1.0 = $500K per incident

Example 2: Partial Data Breach (10% of customer data exfiltrated)
  Asset Value = $100M (estimated value of all customer data)
  Exposure Factor = 0.10 (10% exfiltrated)
  SLE = $100M √ó 0.10 = $10M impact
```

### Annual Rate of Occurrence (ARO)

```
ARO = Number of times a loss event is expected to occur in one year

Ranges from 0 (never) to 1.0+ (annual event or more frequent)

Example: Zero-Day Control Server Compromise
  Historical data: RMM zero-days discovered 4-5 per year industry-wide
  Risk to MSP this size: 200+ annual exploitation attempts
  Success rate (with controls): 0.54% (derived from control effectiveness)
  Expected successful compromises: 200 √ó 0.0054 = 1.08/year
  ARO = 1.08 (expected compromise every 11 months)
  
Example: Ransomware via RMM Agent
  Attempted credential theft: 3.37 per year (threat intelligence)
  Success rate (MFA + strong controls): 2% (98% defeated)
  ARO = 3.37 √ó 0.02 = 0.067 (expected once every 15 years)
```

### Annual Loss Expectancy (ALE)

```
ALE = SLE √ó ARO

Represents expected monetary loss from a risk per year (in $M terms)

Example 1: Control Server Compromise (on-premise with strong controls)
  SLE = $500K (direct incident costs)
  + $10M (regulatory fines - GDPR 4% revenue)
  + $30M (customer churn - 20% lose customers)
  + $10M (legal settlements)
  = $50.5M total impact
  ARO = 0.018 (1 every 56 years)
  ALE = $50.5M √ó 0.018 = $0.909M/year

But adding systemic risks (not just single incident):
  Expected range: $5-10M/year
  Conservative estimate: $7.5M/year

Example 2: Cloud Multi-Tenant Compromise
  SLE = $100M (includes cascade to customer base)
  ARO = 0.337 (1 every 3 years)
  ALE = $100M √ó 0.337 = $33.7M/year
  Conservative estimate: $30M/year
```

### Loss Event Frequency (LEF) - Detailed Breakdown

```
LEF = Threat Event Frequency (TEF) √ó Vulnerability (Vuln)

Where:
  TEF = Number of threat events per year (from threat intelligence)
  Vuln = Probability of successful exploitation (0-1.0)

TEF Component (Threat Intelligence Sourcing):
  - Mandiant M-Trends annual report (APT attack data)
  - Gartner Hype Cycle (RMM platform risks)
  - CISA Alerts (active RMM exploitations)
  - VirusTotal stats (malware targeting RMM platforms)
  - Shodan queries (exposed RMM instances)
  
For 300K-device MSP:
  Global RMM threat actors: 200+/year
  Attacks scaled to MSP size: 3.37/year (2x industry average)
  TEF = 3.37 attempts/year

Vulnerability Component (Control Effectiveness):
  Six control layers:
  1. Perimeter defense (IDSIPS): 60% bypass rate ‚Üí 40% effective
  2. Patching (CIS hardening): 40% unpatched window ‚Üí 60% effective
  3. Access control (MFA+segmentation): 30% bypass ‚Üí 70% effective
  4. Application hardening: 50% exploit reliability ‚Üí 50% effective
  5. Detection (EDR+SIEM): 15% evasion ‚Üí 85% effective
  
  Combined resistance: 1 - (0.60 √ó 0.40 √ó 0.30 √ó 0.50 √ó 0.15) = 99.46%
  Vulnerability: Vuln = 0.0054 = 0.54%
  
LEF = 3.37 √ó 0.0054 = 0.018 successful compromises/year (on-premise)
LEF = 3.37 √ó 0.10 = 0.337 successful compromises/year (cloud)
```

### Residual Risk Calculation

```
Residual Risk = Inherent Risk - (Inherent Risk √ó Control Effectiveness)

Or more precisely:
Residual Risk Score = Inherent Risk Score √ó (1 - Control Effectiveness)

Example:
  Inherent Risk Score (no controls): 95/100 (very high)
  Control Effectiveness (on-premise): 94% (96.4% resistance + 99.4% detection - 2% dual failure)
  Residual Risk = 95 √ó (1 - 0.94) = 95 √ó 0.06 = 5.7/100
  
Interpretation: After implementing on-premise controls, risk reduced from 95 to 5.7 (94% reduction)
This 5.7 residual is acceptable with insurance and operational discipline
```

## Appendix C: Incident Response Playbook (Abbreviated)

### Zero-Day Control Server Compromise Playbook

**Detection Phase (Objective: Identify incident within <24hr)**
1. EDR alert on RMM server (behavioral detection)
2. SIEM correlation of multiple alerts
3. Security Operations Center notified (on-call engineer paged)
4. Incident Commander declared (escalates to CISO)

**Investigation Phase (Objective: Determine scope and impact)**
1. Isolate RMM control server from network (air-gap)
2. Snapshot memory/disk for forensic analysis
3. Review EDR logs (48hr lookback, 7-day history)
4. Review SIEM alerts (identify lateral movement indicators)
5. Check audit logs (identify unauthorized actions)
6. Contact managed customers (brief on incident status)

**Containment Phase (Objective: Stop active exploitation)**
1. Verify all agents still report to RMM (no takeover)
2. Check for suspicious agent updates/commands
3. Review firewall logs for exfiltration attempts
4. Block any suspicious IP addresses at perimeter
5. Rotate all administrative credentials
6. Revoke compromised API tokens

**Eradication Phase (Objective: Remove attacker from environment)**
1. Rebuild control server from clean backup (offline media)
2. Restore database from pre-compromise backup
3. Re-verify all agent integrity (hash check)
4. Change all encryption keys in HSM
5. Update all certificates

**Recovery Phase (Objective: Return to normal operations)**
1. Bring rebuilt RMM server online (isolated network first)
2. Test agent communication with isolated group (100 agents)
3. Expand to 1,000 agents (test group)
4. Expand to 10,000 agents (controlled rollout)
5. Full fleet communication (if no issues detected)
6. Update customer status (incident resolved)

**Post-Incident Phase (Objective: Learn and improve)**
1. Root cause analysis
2. Lessons learned document
3. Process improvements
4. Control enhancements
5. Board briefing on incident and remediation
